Changed Block Tracking vs Resilient Change Tracking

What to consider?

VMware and Hyper-V apply different incremental backup technologies for virtual machines. Compare reliability and convenience of CBT and RCT!

In the world of virtualization full backup is used to prevent data loss in case of emergencies. As long as full backup requires much time and storage, it usually takes place in the beginning and then is followed by incremental backups. They save resources by backing up only those blocks of VMs that have been changed since the last backup or at a certain point in time.

How does the backup product detect changed blocks in VMs? Here incremental backup technologies come in handy, such as Changed Block Tracking (CBT) and Resilient Change Tracking (RCT). CBT was introduced by VMware for vSphere in 2011, whereas RCT was released by Microsoft for Hyper-V in 2016. Both track changes in virtual machines and tell this information to a backup product, both make backup vendors’ work faster and less complicated, as backup applications don’t need to bother with developing change tracking solutions on their own. That’s what CBT and RCT have in common, but what are the peculiarities of these technologies?

Let’s take a closer look at Changed Block Tracking and Resilient Change Tracking mechanisms. CBT scans blocks of VMs, marks changed blocks and stores data changes in a “-ctk.vmdk” file which contains very granular information, because it is based on bitmaps. But in case of a sudden shutdown, power failure or VM migration to another host the file will be lost and a backup product won’t be able to reference it. As for RCT, it eliminates the risk of data loss in unforeseen circumstances by creating 3 files. The first one “.vhdx” is stored in memory, while “.mrt” and “.rct” are saved on disks. It ensures that if crash, power loss or migration happens, files living on disks are still available and changed blocks can be used or reconstructed for new incremental backup. In other words, Microsoft filled in a gap in technology when brought RTC to the field of virtualization.

There is another significant difference. In contrast to Changed Block Tracking, RCT is a native feature. Data protection solutions have to work with vStorage APIs and apply VMkernel filter drivers to access the CBT feature. With Resilient Change Tracking there is no need in a proprietary method because RCT is a native functionality for Windows Server 2016. Built-in features work smoother and require less attention than third-party filters.

Block-level change tracking technologies are essential for efficient incremental backups. Resilient Change Tracking is a younger solution on the market, but it appears to be more reliable in emergency situations and more convenient for data protection products. In comparison to Change Block Tracking, it leaves proprietary filter drives and data loss of changed blocks in the past.
